# Quick Task 260410-s81: Implement Option B - Mobile DANFE via HTML Summary

Short one-liner: Revert the 260410-rvi mobile placeholder card and render the HTML DANFE directly inside the modal iframe on mobile (<768px), keeping desktop behavior byte-identical (original PDF in iframe).

## Objective Recap

Quick task 260410-rvi replaced the modal iframe with a placeholder card on mobile because Chrome/Firefox mobile browsers do not render PDFs inline. The user rejected that UX and asked for "option B": keep the iframe on mobile but feed it HTML instead of PDF. This plan implements that by:

1. Teaching the backend PDF route to return HTML when `?format=html` is passed (when `xmlContent` is available).
2. Simplifying the frontend modal back to a single iframe branch whose `src` is `pdfUrl` on desktop and `${pdfUrl}?format=html` on mobile.

## Tasks Completed

### Task 1: Backend — add `format=html` query param

**Commit:** `d9b555e`
**File:** `src/app/api/invoices/[id]/pdf/route.ts`

Added two variables adjacent to the existing `autoPrint`/`download` declarations:

```ts
const format = url.searchParams.get('format');
const forceHtml = format === 'html' && !!invoice.xmlContent;
```

Gated the `getOriginalIssuedPdf` call with a ternary so that when `forceHtml` is true the OneDrive PDF lookup is skipped entirely and execution falls through to the existing HTML generation path (NFE/CTE/NFSE branches at lines 67-102) and the final `text/html` response at lines 137-144.

Preserved behavior:
- `?download=true` still routes through Puppeteer for a real PDF.
- `?print=true` still sets `no-store` headers on the PDF response.
- Legacy invoices with no `xmlContent` still fall back to OneDrive PDF because `forceHtml` short-circuits to `false`.
- No changes to imports, error handling, logging, or response headers.

Net diff: 11 insertions, 7 deletions (purely structural — the existing `getOriginalIssuedPdf` call was reindented into the ternary).
